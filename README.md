## 🐛 AI 코드 검토로 버그 잡는 법 — 실무에서 검증한 5단계 프레임워크

코드를 짰습니다. 돌렸습니다. 터졌습니다.

에러 메시지를 구글에 복붙했습니다. Stack Overflow를 뒤졌습니다. 30분이 지났습니다.

**이 경험, 한 번이라도 없는 분이 있을까요?**

2025년 지금, 이 과정을 절반 이하로 줄이는 방법이 생겼습니다. AI 코드 검토입니다. 그런데 여기서 중요한 게 있습니다. "AI한테 코드 붙여넣고 고쳐줘" 하는 것과, **체계적인 검토 프레임워크를 쓰는 것**은 결과물이 완전히 다릅니다.

이 글에서는 제가 실무 프로젝트에서 직접 검증한 AI 코드 검토 5단계 프레임워크를 공유합니다. 파이썬 데이터 분석 코드를 기준으로 설명하지만, 원리는 어떤 언어에도 적용됩니다.

---

## 🔍 왜 AI 코드 검토가 강력한가

버그에는 두 종류가 있습니다.

**즉시 터지는 버그**는 사실 쉽습니다. 에러 메시지가 줄 번호까지 알려주니까요. 문제는 **조용히 틀린 버그**입니다.

코드는 실행됩니다. 결과도 나옵니다. 그런데 숫자가 살짝 이상합니다. 데이터프레임 행 수가 예상과 다릅니다. 모델 정확도가 왜인지 모르게 낮습니다.

이런 **논리 오류, 데이터 누수, 잘못된 가정**은 사람 눈으로 찾기가 훨씬 어렵습니다. AI는 바로 이 지점에서 강합니다. 패턴을 기반으로 "이 코드, 의도한 대로 동작하지 않을 가능성이 있습니다"를 짚어주기 때문입니다.

---

## 📋 5단계 AI 코드 검토 프레임워크

### 1단계 — 컨텍스트 먼저 설명하기

가장 흔한 실수가 코드만 던지는 겁니다.

**나쁜 요청 방식:**
"이 코드 버그 찾아줘"

**좋은 요청 방식:**
"판다스로 월별 매출 집계를 하는 코드입니다. 원본 데이터는 일별 거래 기록이고, 최종적으로 월별 합계를 구해야 합니다. 근데 12월 합계가 실제보다 낮게 나오는 것 같습니다. 검토해주세요."

컨텍스트를 주면 AI가 **의도 대비 구현 불일치**를 찾아줍니다. 코드 문법 오류가 아니라 논리 오류를 잡는 핵심입니다.

---

### 2단계 — 검토 관점을 구체적으로 지정하기

AI에게 "전부 다 봐줘"라고 하면 피상적인 답이 옵니다. **검토 관점을 나눠서 요청**하면 훨씬 깊은 분석이 나옵니다.

데이터 분석 코드라면 이 4가지 관점으로 나눠 요청하세요.

**관점 1 — 데이터 무결성 검토**
"결측치 처리가 올바르게 됐는지, 예상치 못한 데이터 손실은 없는지 확인해줘"

**관점 2 — 논리 흐름 검토**
"각 단계의 출력이 다음 단계 입력으로 올바르게 연결되는지 봐줘"

**관점 3 — 엣지 케이스 검토**
"빈 데이터프레임, 음수 값, 중복 행 같은 예외 상황에서 이 코드가 어떻게 동작할지 예측해줘"

**관점 4 — 성능 검토**
"10만 행 이상의 데이터에서 병목이 될 수 있는 부분이 있는지 찾아줘"

---

### 3단계 — 에러 메시지를 통째로 붙여넣기

에러가 났을 때 에러 메시지 일부만 복사하는 분이 많습니다.

**반드시 다음 3가지를 함께 제공하세요.**

첫째, 에러 메시지 전체 (Traceback 포함)
둘째, 에러가 발생한 코드 블록
셋째, 사용 중인 라이브러리 버전

버전 정보가 왜 중요하냐면, pandas 1.x와 2.x는 동작이 다른 함수가 있습니다. AI는 버전 정보 없이는 정확한 원인을 추정하기 어렵습니다. **pip show pandas** 결과를 함께 붙여넣는 습관을 들이세요.

---

### 4단계 — 수정 코드를 그냥 받지 말고 설명을 요구하기

AI가 수정 코드를 줬습니다. 그냥 붙여넣으면 안 됩니다.

**반드시 이 질문을 추가로 하세요:**
"왜 이 부분이 버그였는지, 그리고 수정 코드가 어떻게 그 문제를 해결하는지 설명해줘"

이유가 두 가지입니다.

하나, **AI도 틀립니다.** 설명을 들으면 논리가 맞는지 직접 검증할 수 있습니다.

둘, **같은 실수를 반복하지 않게 됩니다.** 수정 코드만 받으면 왜 틀렸는지 모르고 넘어갑니다. 설명을 들으면 그 패턴이 내 것이 됩니다. 이것이 AI 코드 검토가 단순 디버깅을 넘어 **학습 도구**가 되는 지점입니다.

---

### 5단계 — 재현 가능한 최소 예제로 좁히기

버그를 못 찾겠을 때 쓰는 최후의 방법입니다.

전체 코드가 200줄이라면, 버그가 있는 부분만 **20줄짜리 독립 실행 코드**로 만들어서 요청하세요.

이 과정 자체가 버그를 찾는 데 도움이 됩니다. 최소 예제를 만들다 보면 "아, 여기서 이미 뭔가 이상하네"를 발견하는 경우가 절반 이상입니다. 남은 절반은 AI가 훨씬 정확한 답을 줍니다. 관련 없는 코드가 없으니까요.

## ⚠️ AI 코드 검토의 한계 — 이것만큼은 믿지 마세요

AI 코드 검토를 맹신하면 안 되는 영역이 있습니다.

**도메인 특화 비즈니스 로직**은 AI가 모릅니다. "이 업종에서 매출 집계는 반품 처리일로부터 3영업일 후 확정"같은 규칙은 코드에 명시되지 않으면 AI는 검토할 수 없습니다.

**데이터 품질 문제**도 마찬가지입니다. 코드는 맞는데 원본 데이터가 잘못됐다면 AI 코드 검토로는 발견이 안 됩니다. 데이터 프로파일링은 별도로 해야 합니다.

**보안 취약점**은 전문 도구(Bandit, Semgrep 등)와 병행하는 것을 권장합니다. AI가 잡아주는 경우도 있지만 체계적이지 않습니다.

## 🎯 오늘 바로 써먹는 프롬프트 3가지

**버그 찾기용:**
"다음은 [목적] 코드입니다. [증상]이 발생합니다. 논리 오류, 데이터 처리 순서 문제, 엣지 케이스 3가지 관점에서 검토해주세요. [코드]"

**코드 리뷰용:**
"이 코드가 동작은 하지만 더 안전하고 효율적으로 개선할 수 있는 부분을 찾아주세요. 각 제안에 이유를 함께 설명해주세요. [코드]"

**에러 해결용:**
"다음 에러가 발생했습니다. [Traceback 전체]. 원인과 해결 방법을 알려주고, 같은 에러가 다시 발생하지 않도록 코드를 어떻게 수정해야 하는지 설명해주세요."

## 📌 핵심 3줄 요약

AI 코드 검토는 **코드만 붙여넣기**가 아니라 컨텍스트 + 관점 + 에러 전체를 함께 제공할 때 제대로 동작합니다.

수정 코드를 받으면 **왜 틀렸는지 설명을 요구**하세요. 이것이 실력이 늘어나는 방식입니다.

도메인 로직과 데이터 품질은 AI 검토의 **사각지대**입니다. 이 부분은 여전히 사람이 책임져야 합니다.
